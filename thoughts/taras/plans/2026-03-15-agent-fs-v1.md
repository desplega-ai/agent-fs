# agent-fs v1 Implementation Plan

## Overview

Build agent-fs v1: an agent-first filesystem API backed by S3. The product exposes POSIX-named file operations, semantic search, auto-versioning, and RBAC — via a REST API, CLI, and MCP server. This plan covers local mode only (daemon + CLI + MCP, BYO S3).

## Current State Analysis

Greenfield project. The repository contains only a `thoughts/` directory with:
- Brainstorm document: `thoughts/taras/brainstorms/2026-03-14-agent-fs.md`
- Research: embeddings/qmd, git-versioning-on-S3, competitive landscape, pricing analysis

No code exists yet.

### Key Discoveries:
- Drizzle ORM works with `bun:sqlite` via `drizzle-orm/bun-sqlite`. FTS5/sqlite-vec require raw SQL via `db.$client` — Drizzle doesn't support virtual tables natively.
- macOS requires `Database.setCustomSQLite()` pointing to Homebrew's SQLite before loading extensions (Apple's bundled SQLite lacks extension support).
- MCP SDK (`@modelcontextprotocol/sdk` v1.x) supports stdio + Streamable HTTP transports. Tools defined with Zod schemas, can be registered programmatically. All registrations must happen before `server.connect()`.
- `node-llama-cpp` officially supports Bun. API: `getLlama() → loadModel() → createEmbeddingContext() → getEmbeddingFor()`.
- sqlite-vec vectors bind as `Float32Array`. KNN queries use `WHERE embedding MATCH vec_f32(?) ORDER BY distance LIMIT N`.
- Hono is the recommended HTTP framework for Bun (Web Standards-based, ~14KB, multi-runtime).

## Desired End State

A working local-mode agent-fs with:
1. `agentfs init` → configure S3 + embedding provider
2. `agentfs daemon start` → starts HTTP API on localhost:7433
3. `agentfs ls /reports/` → lists files (JSON output)
4. `agentfs write /reports/q1.md < content` → writes file, auto-versions
5. `agentfs search "revenue analysis"` → semantic search across files
6. `agentfs mcp` → stdio MCP server for Claude Code / agent harnesses
7. All operations enforce RBAC per drive

### Local S3 (MinIO) Strategy

For local development, `agentfs init` manages a MinIO Docker container automatically:

**Init decision tree:**
```
agentfs init
├─ S3 provider?
│  ├─ Local (MinIO Docker)
│  │  → docker pull minio/minio (if not cached)
│  │  → docker run -d --name agentfs-minio -p 9000:9000 minio/minio server /data
│  │  → Generate random access key + secret key via MinIO admin API
│  │  → Write credentials to ~/.agentfs/config.json
│  │  → Create default bucket ("agentfs")
│  │  → Store container ID in ~/.agentfs/config.json (minio.containerId)
│  │  → Confirmation prompt (bypass with -y)
│  └─ Bring your own S3
│     → Prompt for: provider (AWS/R2/Tigris/custom), endpoint, bucket, access key, secret key
├─ Embedding provider?
│  ├─ Local (embeddinggemma-300M) → auto-download model (~329MB)
│  ├─ OpenAI → prompt for API key
│  └─ Gemini → prompt for API key
```

**Shortcuts:**
- `agentfs init --local` → MinIO Docker + local embeddings, no prompts (`-y` implicit)
- `agentfs init -y` → accept all defaults without confirmation (agent-friendly)

**Lifecycle integration:**
- `agentfs daemon start` → also starts the MinIO container if it was stopped
- `agentfs daemon stop` → also stops the MinIO container (not removed, data persists)
- `agentfs daemon status` → shows both API server and MinIO container status
- MinIO container uses a Docker volume for persistence (`agentfs-minio-data`)
- Container ID stored in `~/.agentfs/config.json` under `minio.containerId`

## Implementation Approach

**Monorepo with Bun workspaces.** Four packages:
- `packages/core` — business logic, S3 client, database, ops engine, search, identity
- `packages/server` — Hono HTTP server (daemon)
- `packages/cli` — Commander.js CLI
- `packages/mcp` — MCP server (stdio + HTTP)

Core is the dependency of all other packages. Server, CLI, and MCP are thin wrappers over core.

**Build order:** Infrastructure → storage engine → search → identity/permissions → HTTP API → CLI → MCP. Each phase is independently testable.

**Design note — Op registry vs OpenAPI spec:** The brainstorm specified "REST API (OpenAPI spec) as foundation, auto-generated MCP + CLI." This plan uses a Zod-based op registry instead, which is more natural for TypeScript (runtime validation + type inference in one pass). An OpenAPI spec can optionally be generated FROM the op registry using `@hono/zod-openapi` for documentation purposes, but the op registry is the source of truth.

**Database migrations:** Drizzle migrations run automatically on daemon startup (`migrate()` called before `Bun.serve()`). Schema changes between versions are handled transparently — users just upgrade the package and restart the daemon.

**npm packages:** `@agentfs/core`, `@agentfs/server`, `@agentfs/mcp` are internal workspace packages. The CLI is published as `agentfs` (no scope) for ergonomics (`bun install -g agentfs`).

**Tech stack:**
| Concern | Package |
|---------|---------|
| Runtime | Bun |
| HTTP framework | `hono` |
| ORM | `drizzle-orm` + `bun:sqlite` |
| Migrations | `drizzle-kit` (generate) + custom runner |
| S3 client | `@aws-sdk/client-s3` |
| Vector search | `sqlite-vec` (extension) |
| Full-text search | SQLite FTS5 (built-in) |
| Local embeddings | `node-llama-cpp` + embeddinggemma-300M |
| Cloud embeddings | `openai`, `@google/genai` |
| MCP SDK | `@modelcontextprotocol/sdk` + `zod` |
| CLI framework | `commander` |
| Text chunking | `chonkie` (RecursiveChunker) |
| Diffing | `diff` (jsdiff) |
| Distribution | npm `bin` field + `bun build --compile` |

`raw.ts` — Raw SQL for virtual tables (FTS5 + sqlite-vec):
```sql
-- FTS5 full-text index (internal content storage — simpler, no external content table needed)
CREATE VIRTUAL TABLE IF NOT EXISTS files_fts USING fts5(
  path, content, drive_id UNINDEXED
);

-- sqlite-vec vector index for semantic search (768 dimensions)
CREATE VIRTUAL TABLE IF NOT EXISTS chunk_vectors USING vec0(
  chunk_id INTEGER PRIMARY KEY,
  embedding float[768]
);
```
**Note on 768 dimensions:** This is the universal sweet spot across all providers:
- **Local** (embeddinggemma-300M): outputs 768 natively
- **OpenAI** (text-embedding-3-small): supports MRL truncation to 768 via `dimensions: 768` param
- **Gemini** (gemini-embedding-001): recommended dimension is 768 (truncated from 3072)
All providers produce compatible 768-d vectors. Switching providers requires re-embedding (same dimensions, different vector spaces).

`index.ts` — Database connection with extension loading:
- On macOS: `Database.setCustomSQLite()` to Homebrew SQLite before creating instance
- Load sqlite-vec extension via `sqliteVec.load(db)`
- Create Drizzle instance wrapping the raw database
- Run virtual table creation SQL from `raw.ts`

`migrate.ts` — Migration runner:
- Uses `drizzle-orm/bun-sqlite/migrator`
- Also runs raw SQL for virtual tables
- Idempotent (safe to run multiple times)

#### 5. S3 Client Abstraction
**Files**: `packages/core/src/s3/client.ts`, `packages/core/src/s3/index.ts`
**Changes**:
- Wrapper around `@aws-sdk/client-s3` configured from `config.json`
- **Why not Bun's built-in S3?** Bun's native `Bun.s3` is faster but missing `copyObject` (needed for `cp`/`mv`) and `listObjectVersions` (needed for versioning). Both are critical. Stick with AWS SDK which covers all operations.
- Methods: `putObject(key, body, metadata?)`, `getObject(key, versionId?)`, `deleteObject(key)`, `copyObject(from, to)`, `listObjects(prefix)`, `headObject(key)`, `listObjectVersions(key)`, `checkVersioningEnabled(bucket)`
- All methods take the S3 key directly (org/drive prefix handled by caller)
- Returns typed results, not raw AWS SDK responses
- **S3 versioning detection:** On init, check if bucket versioning is enabled (`GetBucketVersioning`). If enabled, store `s3.versioningEnabled: true` in config. If not, attempt to enable it (`PutBucketVersioning`). If that fails (e.g., insufficient IAM permissions), warn the user and set `s3.versioningEnabled: false`. When versioning is disabled:
  - `write`/`edit`/`append` still work but don't record `s3VersionId` (SQLite-only versioning via content snapshots)
  - `log` works (from SQLite records) but `diff` falls back to stored `diffSummary` only (can't fetch old versions from S3)
  - `revert` is disabled (returns error: "S3 versioning required for revert")

## Phase 2: File Operations & Versioning Engine

### Overview
Build the core operations engine. Every file op is a standalone async function that takes an `OpContext` and typed params, talks to S3 + SQLite, and returns a typed result. Every mutation auto-creates a version record.

### Changes Required:

#### 1. Op Context & Types
**File**: `packages/core/src/ops/types.ts`
**Changes**:
- `OpContext` type: `{ db, s3, bucket, orgId, driveId, userId }`
- Result types for each op (matching the brainstorm's API response shapes)
- `OpResult<T>` wrapper with success/error union

#### 2. Version Helper
**File**: `packages/core/src/ops/versioning.ts`
**Changes**:
- `createVersion(ctx, { path, s3VersionId, operation, message?, diffSummary?, size, etag })` — inserts into `file_versions`, updates `files` metadata
- `getNextVersion(ctx, path)` — returns next monotonic version number for a file
- `getVersionS3Key(orgId, driveId, path)` — computes the S3 object key: `<orgId>/drives/<driveId>/<path>`

#### 3. File Operations
**Files**: `packages/core/src/ops/write.ts`, `cat.ts`, `ls.ts`, `stat.ts`, `rm.ts`, `mv.ts`, `cp.ts`, `head.ts`, `tail.ts`, `mkdir.ts`, `edit.ts`, `append.ts`

**Atomicity strategy:** S3 is the source of truth. All mutating ops follow this order:
1. Perform S3 operation first (PutObject/CopyObject/DeleteObject)
2. On S3 success → update SQLite metadata in a transaction (version record + file metadata)
3. On SQLite failure → log error, mark file for metadata repair (file exists in S3 but metadata is stale)
4. For compound ops like `mv` (copy + delete): copy first → update SQLite → delete original. Failure at any step leaves recoverable state.
5. A background repair job can detect S3/SQLite drift by comparing `listObjects` against the `files` table.

Each op follows the same pattern:
```typescript
export async function write(ctx: OpContext, params: WriteParams): Promise<WriteResult> {
  // 1. Validate params
  // 2. Compute S3 key
  // 3. S3 operation (PutObject)
  // 4. Create version record in SQLite
  // 5. Update file metadata in SQLite
  // 6. Trigger async indexing (FTS5 sync, embedding async)
  // 7. Return typed result
}
```

Specific op details:

**Large file strategy:** For operations that read file content (`cat`, `head`, `tail`, `edit`, `append`), stream the S3 response to a temp file (`Bun.write(tempPath, s3Response.Body)`) instead of holding it in memory. Read specific line ranges from the temp file. Clean up temp file after the op completes. This prevents OOM for large files.

**`write`**: PutObject to S3 → create version (operation="write") → upsert file metadata. Returns `{ version, path, size }`.

**`cat`**: GetObject from S3 → split into lines → apply offset/limit → return with truncation notice if needed. Default limit: 200 lines. Returns `{ content, totalLines, truncated }`.

**`edit`**: GetObject → verify `old_string` exists exactly once (throw `EditConflictError` if not found OR if found multiple times — same behavior as Claude's Edit tool) → replace → PutObject → create version with `diffSummary: JSON.stringify({ old: old_string, new: new_string })`. Returns `{ version, path, changes: 1 }`.

**`ls`**: ListObjectsV2 with prefix → join with SQLite file metadata → return entries with name, type, size, author, modified. Returns `{ entries: LsEntry[] }`.

**`stat`**: HeadObject + SQLite metadata query. Returns full metadata without content.

**`rm`**: DeleteObject (creates S3 delete marker) → create version (operation="delete") → mark file as `isDeleted=true`. Returns `{ path, deleted: true }`.

**`mv`**: CopyObject + DeleteObject → create version on new path (operation="write", message="moved from ...") → mark old path deleted. Returns `{ from, to, version }`.

**`cp`**: CopyObject → create version on new path → copy file metadata. Returns `{ from, to, version }`.

**`head`**: Calls `cat` with `limit=N` (default 20). Returns same shape as `cat`.

**`tail`**: GetObject → split lines → take last N. Returns same shape as `cat`.

**`mkdir`**: PutObject with empty body at `path/` (S3 prefix convention). Returns `{ path }`.

**`append`**: GetObject → append content → PutObject → create version (operation="append"). Returns `{ version, size }`.

#### 4. Versioning Operations
**Files**: `packages/core/src/ops/log.ts`, `diff.ts`, `revert.ts`

**`log`**: Query `file_versions` WHERE path + driveId, ORDER BY version DESC, LIMIT. Returns `{ versions: VersionEntry[] }` with version, author, timestamp, operation, message, diffSummary.

**`diff`**: If both versions are `edit` operations with stored diffSummary, return the stored diffs directly. Otherwise, GetObject for both versions → compute diff using `jsdiff.structuredPatch()`. Returns `{ changes: DiffChange[] }`.

**`revert`**: GetObject with versionId from target version → PutObject as new current → create version (operation="revert", message=`Reverted to version ${n}`). Returns `{ version, revertedTo }`.

#### 5. Activity Feed
**File**: `packages/core/src/ops/recent.ts`
**Changes**:
- `recent(ctx, { path?, since?, limit? })` — query `file_versions` with optional path prefix filter and time range
- Returns `{ entries: RecentEntry[] }` — same as version entries but across all files in the drive

#### 6. Op Registry

**Changes**:

- `dispatchOp(ctx, opName, params)` — validates params with Zod, calls handler, returns result
- This becomes the single entry point used by server, CLI, and MCP

#### 7. Tests
**Files**: `packages/core/src/ops/__tests__/*.test.ts`
**Changes**:

- Test versioning: write → edit → log → verify version chain

- Test edit conflict: write → edit with wrong old_string → expect EditConflictError

- Config: ~900 tokens per chunk, 15% overlap (~128 tokens)

#### 8. Embedding Pipeline
**File**: `packages/core/src/search/pipeline.ts`
**Changes**:
- `indexFileEmbeddings(ctx, { path, driveId, content })` — async pipeline:
  1. Chunk content via `chunkContent()`
  2. Delete old chunks + vectors for this file
  3. Insert new chunks into `content_chunks`
  4. Embed all chunks via configured provider
  5. Insert vectors into `chunk_vectors` (sqlite-vec)
  6. Update file's `embeddingStatus` to "indexed"
- Error handling: on failure, set `embeddingStatus` to "failed", log error, don't block
- Called asynchronously by write/edit/append ops (non-blocking)
- **Async mechanism:** Fire-and-forget `Promise` with a concurrency semaphore (max 2 concurrent embedding jobs). Pattern:
  ```typescript
  const semaphore = new Semaphore(2); // simple counter-based semaphore
  function scheduleEmbedding(ctx, file) {
    semaphore.acquire().then(async (release) => {
      try { await indexFileEmbeddings(ctx, file); }
      catch (e) { logger.error("Embedding failed", { path: file.path, error: e }); }
      finally { release(); }
    });
  }
  ```
  This prevents overwhelming the embedding provider (especially cloud APIs) while keeping writes non-blocking. For local embeddings (node-llama-cpp), the semaphore also prevents multiple concurrent model loads.

**`search`**: Embeds query → sqlite-vec KNN query → join with file metadata → return results with path, score, snippet, author, modified. Returns `{ results: SearchResult[] }`.

- A `generateCommands(program, apiClient)` function loops over the op registry and creates a Commander command for each op:

```typescript
import { opRegistry } from "@agentfs/core";
import { Command } from "commander";

export function registerOpCommands(program: Command, apiClient: ApiClient) {
  for (const [name, def] of Object.entries(opRegistry)) {
    const cmd = new Command(name).description(def.description);

    // Map Zod schema fields to Commander arguments/options
    for (const [param, schema] of Object.entries(def.schema)) {
      if (def.positionalArgs?.includes(param)) {
        cmd.argument(`<${param}>`, schema.description);
      } else {
        cmd.option(`--${param} <value>`, schema.description);
      }
    }

    cmd.action(async (...args) => {
      const result = await apiClient.callOp(name, extractParams(args, def));
      console.log(JSON.stringify(result, null, 2));
    });

    program.addCommand(cmd);
  }
}
```

## Distribution

### npm Package
- `npm install -g agentfs` or `bun install -g agentfs`
- `bin` field in `packages/cli/package.json` → `agentfs` command available globally
- Shebang: `#!/usr/bin/env bun`

### Standalone Binaries — Native Extension Caveat

`bun build --compile` bundles JS/TS into a single executable but does NOT embed native C/C++ addons (`sqlite-vec`, `node-llama-cpp`). These are resolved at install time per platform via npm's `optionalDependencies` mechanism.

**Consequence:** Standalone binaries (`bun build --compile`) will NOT work out of the box for agent-fs because native extensions won't be found at runtime.

**Distribution strategy (revised):**

1. **npm (primary)**: `bun install -g agentfs` — native deps resolved per platform at install time. This is the recommended install method.
2. **Docker image**: `docker run agentfs/agentfs` — all native deps pre-compiled in the image. Best for servers/CI.
3. **Standalone binary (future)**: Requires either:
   - Bundling prebuilt `.node` files alongside the binary per platform
   - Using platform-specific npm packages as `optionalDependencies` (Sentry's pattern)
   - Deferred to post-v1 when the distribution needs are clearer

### GitHub CI Action
**File**: `.github/workflows/release.yml`
- Triggered on version tag push (`v*`)
- Publish to npm registry (`npm publish`)
- Build and push Docker image to GitHub Container Registry
- Future: matrix build for standalone binaries once native addon bundling is solved

### Install Methods
1. **npm/bun** (primary): `bun install -g agentfs`
2. **Docker**: `docker pull ghcr.io/agentfs/agentfs`
3. **Standalone binary** (future, post-v1)
