# agent-fs — Agent-first Filesystem API backed by S3 — Brainstorm

## Context

Taras wants to build a filesystem API designed for AI agents (agent-first), inspired by agentmail but for filesystems instead of email. The core idea: agents should be able to access a shared filesystem the way a human team uses Google Drive — with permissions, discoverability, and familiar operations.

**Key initial ideas:**
- Exposes both an MCP server and a CLI (same underlying tools)
- Built on top of S3 as the storage backend
- Granular permissions (agent-level, path-level)
- S3-like path structure for simplicity
- POSIX-compliant operations (ls, cat, wc, grep, etc.)
- Auto-embedding for semantic search
- Human UI for visibility/management, but agents are the primary consumers

**Inspiration:** agentmail (agent-first email) → agent-fs (agent-first filesystem)

## Exploration

### Q: Who are the primary agents using agent-fs?
The target is external agent harnesses — Claude Code, Codex, Pi, etc. Any agent that can use MCPs or skills to interact with a CLI. Not limited to Taras's own swarm. Future possibility: mounting FS support (like Archil does) so agents could interact with it as a local filesystem.

**Insights:** This means agent-fs is a **product/service**, not just an internal tool. It needs to work with any MCP-capable agent, which means the API surface must be simple and self-descriptive. Authentication and multi-tenancy become first-class concerns from day one. The FUSE/mount idea is interesting as a future layer but shouldn't drive the initial design.

### Q: What core problem does this solve that existing solutions don't?
All of the above — it's a compound problem:
1. **No shared persistence** — agents lose context between sessions, no durable shared space
2. **Ad-hoc workarounds** — people use git repos, raw S3, local files — messy and unstructured
3. **Permissions are missing** — raw S3 and git don't have agent-level access control
4. **Discoverability** — agents can't search/find relevant files across a shared space
5. **Shared knowledge + agent-first DX** — the real gap is that nothing is designed for agents as first-class consumers. The DevEx from the agent's perspective needs to be "clean AF."

**Insights:** The value prop isn't any single feature — it's the combination packaged as a coherent agent-first experience. The analogy holds: Google Drive didn't invent file storage, it made it collaborative and accessible. agent-fs doesn't invent S3, it makes it agent-collaborative and agent-accessible. The "clean AF DX" requirement suggests the MCP tool definitions and CLI commands need to be extremely intuitive — an agent should be able to figure out what to do with minimal system prompt guidance.

### Q: What's the identity model? How are agents and humans identified?
Same as Google Drive, but agents and humans are treated as equals — they're "the same" type of entity. Research agentmail.to's positioning for reference on how they handle this.

**Insights:** This is a key design decision: no separate "agent" vs "human" identity class. Both are principals with the same capabilities. This simplifies the permission model and avoids the trap of second-class agent citizens. The org/team hierarchy likely mirrors Drive: Organization → Team/Shared Drive → Members (humans or agents).

### Q: What does a "file" look like in agent-fs?
Direct mapping to S3 objects. Taras likes S3's path-based approach because it enables easy hierarchy, listing, prefix matching, etc. (Hive partitioning as an analogy — `year=2026/month=03/data.parquet`). The path IS the organization structure.

**Insights:** This is clean — S3 paths as the canonical file model means no separate "folder" concept, just prefix conventions. Metadata can live in S3 object metadata + a thin metadata layer for search/embedding. The hive partitioning reference suggests agents could use convention-based paths like `org/project/type/file` and get powerful listing/filtering for free from the path structure alone.

### Q: Literal POSIX tool names or POSIX-inspired abstractions?
Literal POSIX names — `ls`, `cat`, `grep`, `mv`, `cp`, `rm`, etc. Agents already know these. Reference: Archil provides full POSIX-compatible file access to S3, translating POSIX operations into S3 protocol even when not natively supported (file locks, random writes, renames, symlinks, permissions).

**Insights:** This is a strong DX choice. Every LLM has deep training data on POSIX commands — using literal names means zero learning curve for agents. The Archil reference is helpful: they've already solved the "POSIX over S3" translation problem at the storage layer. The question for agent-fs is whether to go that deep (FUSE mount, random writes, locks) or stay at a higher level where the POSIX names are the API but the semantics are slightly simplified (e.g., no file locks in v1, no random writes — just full read/write). Key tension: POSIX compliance depth vs. implementation complexity.

### Q: Full POSIX compliance or POSIX-named simplified API?
**API now, FUSE later.** Start as a clean MCP/CLI service with POSIX command names but simplified semantics. No file locks, no random byte-offset writes, no symlinks in v1. FUSE/mount support can be a future layer on top.

**Insights:** This is the right call for a product. Ship the API that agents can use immediately via MCP, defer the hard FUSE engineering. The API layer becomes the stable contract — FUSE just becomes another client. This also means the v1 operations are probably: `ls`, `cat`, `cp`, `mv`, `rm`, `mkdir`, `touch`, `grep`, `wc`, `find`, `head`, `tail`, `write/tee`. Full file read/write, not byte-level random access.

### Q: What content types should auto-embedding support?
Start with text + structured data (markdown, JSON, YAML, code, logs, CSV, Parquet, tabular data). Expand to multimodal (images, PDFs, audio) later.

**Insights:** Text + structured data covers the vast majority of what agents produce and consume today. Structured data (CSV, Parquet) is interesting because it means embedding might need to be smarter — embed column names + sample rows, not raw bytes. This also implies agent-fs could have some content-type awareness: it knows a `.csv` should be embedded differently than a `.md`. The embedding pipeline becomes: detect content type → extract text/structure → chunk → embed → index.

### Q: What permissions model?
**RBAC** — predefined roles (viewer, editor, admin) assigned per drive, like Google Drive.

**Insights:** RBAC is the right abstraction for a product. It's familiar to humans (they already think in Drive terms), and it's simple enough for agents to reason about ("I have editor access to /reports/"). Likely roles: `viewer` (read, list, search), `editor` (+ write, delete, move), `admin` (+ manage permissions, create drives). The drive becomes the scope for role assignment.

### Q: Should there be a drive concept?
**Drives as sugar over prefixes.** Under the hood it's flat S3 prefixes (`org/drive/path`), but "drives" are a named convenience layer — a prefix with a default RBAC policy. Agents see clean paths, humans see manageable containers in the UI. Cross-drive ops remain simple because it's all one namespace.

**Insights:** This avoids the complexity of separate namespace isolation while giving humans the familiar "Shared Drive" mental model. A drive is really just: `{ name, prefix, default_roles, members }`. Creating a drive = creating a named prefix + default ACL. This also maps cleanly to S3 bucket/prefix structure: one bucket per org, prefixes per drive.

### Q: Deployment model?
**OSS by default (BYO S3), SaaS on top later.** The core product is open source — users bring their own S3 bucket. A hosted SaaS can be built on top as a managed offering.

**Insights:** This is the agentmail playbook and it's smart: OSS builds trust and adoption in the agent ecosystem, SaaS monetizes convenience. It also means the architecture must cleanly separate the S3 backend configuration from the API/permission layer. The core should work with any S3-compatible storage (AWS S3, R2, MinIO, etc.), not just AWS.

### Q: Tech stack?
**Deferred** — Taras correctly pointed out we should understand the architectural requirements before choosing tech. Tech follows architecture, not the other way around.

### Q: What sits between the agent and S3?
Need to figure out: where do embeddings live, is S3 access always direct or cached, what's the data flow. These are the real architectural questions.

### Q: Where do embeddings live?
**Embedded vector DB** (e.g., SQLite + sqlite-vec) — zero external dependencies, ships with the server. Easiest to run locally for the OSS use case. Also wants deployment templates for production setups.

**Insights:** Embedded DB is the right default for OSS. SQLite-vec or DuckDB with vector support means `agent-fs` is a single binary/process — no Postgres, no Qdrant to set up. For the SaaS tier, this can be swapped to pgvector or a managed vector DB. Deployment templates (Docker Compose, Fly.io, Railway, etc.) would lower the barrier to going from "running locally" to "running in the cloud."

### Q: S3 caching strategy — what's the fastest approach?
**Hybrid: direct S3 for single-file ops, local index for multi-file ops.**

Architecture:
- **S3** = source of truth for file content
- **SQLite metadata DB** = permissions, file metadata, path index, vector embeddings (sqlite-vec)
- **SQLite FTS5 content index** = full-text search for grep/find operations
- Single-file reads (`cat`, `head`, `tail`) → S3 direct (~50-200ms, acceptable for agents)
- Multi-file queries (`grep`, `find`, `wc`) → local FTS5/metadata index (fast, sub-10ms)
- Writes → S3 direct + async index update (metadata, FTS5, embeddings)

**Insights:** This is elegant because `grep` becomes *faster than real grep* — it hits FTS5 instead of sequential file reads. The content index is built async on writes, so writes aren't blocked. The metadata DB serves double duty: permissions + search index + embeddings. All in SQLite = single-process, zero external deps. The only consistency trade-off: a freshly written file might take a moment to appear in `grep` results (eventual consistency on the index), but `cat` on the exact path returns immediately from S3.

### Q: Claude's perspective as a user of agent-fs

**What I (Claude in Code) would love:**

1. **Semantic search across shared knowledge** — This is the killer feature, hands down. Right now when I'm dropped into a task, I can only search local files. If 5 agents have been researching different aspects of a project and writing their findings, I want to do `search "how does auth work in this system"` and get back the relevant docs from all of them. This alone would be a game-changer.

2. **Zero-ceremony reads/writes** — `cat reports/q1.md` should return content, period. No pagination tokens, no session management, no thinking about auth. The MCP connection handles auth once, every operation after that just works.

3. **Provenance on files** — When I find a file, I want to know WHO wrote it (agent? human?) and WHEN. `ls -l /shared/research/` showing `agent-researcher-3  2h ago  auth-analysis.md` helps me judge trustworthiness and relevance.

4. **Smart large-file handling** — If I `cat` a 10MB file, I'm going to destroy my context window. I'd want the tool to either: (a) return a warning + file size and suggest `head`/`tail`, or (b) have a built-in line/byte limit with truncation notice. This is *critical* for agent DX.

**What would frustrate me:**

1. **Write-then-search inconsistency** — If I write a file and immediately `grep` for it and get nothing, I'd be confused and might waste cycles debugging. Either make indexing synchronous for the writing agent, or return a clear "indexing in progress" signal.

2. **Opaque permission errors** — "Permission denied" is useless. "You have `viewer` role on `/reports/`. To write, ask an admin to grant `editor` access." is actionable. Agents can't fix permissions themselves, but they CAN communicate the problem clearly to the human.

3. **Path ambiguity** — If I have to remember the full drive prefix every time (`/org-123/drive-abc/reports/q1.md`), that's miserable. A working directory concept or drive-scoped context (connected to drive X, so `/reports/q1.md` resolves correctly) would be essential.

4. **No file size metadata in `ls`** — Before I `cat` something, I want to know how big it is. `ls` should return size, type, last modified, and author by default.

**What I think is missing from the design so far:**

1. **`write` / `tee` tool** — We listed read operations but haven't talked about the write interface. For me, the ideal is: `write /path/to/file.md <content>` — a single tool that creates or overwrites. Also `append /path/to/file.md <content>` for log-style files.

2. **`stat`** — File metadata without content. Size, type, permissions, author, timestamps, embedding status. Cheap call I'd use before deciding to `cat`.

3. **`diff`** — If a file was updated, I want to see what changed. Version history or at minimum last-N-versions diff would be incredibly useful for agents collaborating on shared documents.

4. **Atomic concerns** — Two agents writing the same file simultaneously. Last-write-wins is fine for v1, but the write response should include a version/etag so agents CAN do optimistic concurrency if they want.

**Honest concern about the architecture:**
The SQLite-based approach is great for single-node OSS. But if two agent-fs instances need to share the same data (e.g., in a multi-region SaaS deployment), SQLite becomes a bottleneck. Worth flagging now even if it's a v2 problem — the abstraction layer should exist so SQLite can be swapped for Postgres+pgvector later without rewriting the API layer.

### Q: Write semantics and auto-versioning

**Key insight from Taras:** Agent tool signatures carry semantic meaning. Claude's `Edit(path, old_string, new_string)` and `Write(path, content)` already encode intent. If agent-fs captures this, you get rich auto-versioning for free.

**Proposed write operations:**
- `write(path, content)` → creates new version, stores full content. Diff computed against previous version.
- `edit(path, old_string, new_string)` → creates new version, stores the semantic patch + resulting content. The diff IS the operation itself.
- `append(path, content)` → adds to end of file, new version.

**What gets stored per version (metadata DB):**
- `version_id` (maps to S3 version ID)
- `author` (agent/human identity)
- `timestamp`
- `operation_type` (write | edit | append)
- `diff_summary` (for edits: the old_string/new_string; for writes: computed diff)
- Optional `message` (like a commit message — agents could pass context about WHY)

**What this enables:**
- `log(path)` → version history (who, when, what operation, diff summary) — like `git log` for a file
- `diff(path, v1?, v2?)` → changes between versions
- `revert(path, version?)` → rollback to a previous version
- `blame(path)` → who wrote each section (for text files)

**Insights:** This is like Google Docs version history meets git, but agent-native. S3 versioning handles the storage for free — no separate version store needed. The metadata DB adds the semantic layer. The `edit` operation is particularly powerful because agents naturally produce patches, not full rewrites — capturing that intent means diffs are human-readable, not byte-level. This could be a major differentiator: no other agent filesystem gives you "who changed what and why" out of the box.

Optional `message` parameter on writes — agents can add commit-message-style context but aren't forced to.

### Q: Search UX for agents
**Rich results:** path + relevance score + matching snippet + author + timestamp + file type. Give the agent maximum context to decide what to read next without wasting tool calls on `cat` for irrelevant files.

### Q: Git as the caching/versioning layer?
**Taras's idea:** Use a local git repo as the cache layer. Auto-commit every X time. S3 is the "main branch" source of truth. Conflicts become visible as git merge conflicts.

**Why this is compelling:**
- Git gives you versioning, diffing, blame, log **FOR FREE** — no need to build any of it
- Local git repo = instant reads, instant writes (no S3 latency for anything)
- `grep`, `cat`, `find`, `diff`, `blame`, `log` all just work on the local working tree using battle-tested git tooling
- The `edit(old_string, new_string)` operation maps naturally to git patches
- S3 push/pull is async — agents get local-speed performance
- Conflict detection comes free with git merge semantics

**Why it's risky:**
- **Git doesn't scale for large files** — but we said text + structured data first, so this might be fine for v1
- **Conflict resolution for agents** — if two agents edit the same file and git detects a conflict, who resolves it? Agents aren't great at merge conflict resolution. Need a strategy (last-write-wins? lock-on-edit? conflict notification?)
- **Git repo bloat** — auto-commits every X seconds on a busy filesystem could create massive git history. Would need periodic squash/gc.
- **S3 ↔ git sync is non-trivial** — bidirectional sync with conflict detection is a real engineering problem. It's essentially building a distributed system.
- **Multiple agent-fs instances** — if two instances serve different agents, each has its own git repo, and they need to stay in sync via S3. This is distributed git, which is hard.

**Claude's honest take:**
The git-like SEMANTICS (versions, diffs, blame) are brilliant and should absolutely be in the product. The question is whether to use **actual git as the implementation** or **build the same semantics on top of S3 versioning + SQLite**.

Using actual git is high-reward (everything works out of the box) but high-risk (sync complexity, conflict resolution, scalability). S3 versioning + SQLite is more work upfront but avoids the distributed system pitfalls.

A possible middle ground: **use git locally as a read cache / working tree, but S3 remains the write target**. Reads come from the local git-backed cache. Writes go to S3 first, then the local git repo pulls the change. This avoids bidirectional sync — it's always S3 → git (one direction).

**Further exploration — git as the storage backbone:**
Taras's deeper idea: git IS the backbone, even in S3. Agents get branches on connect. Work in isolation, merge back to main. This is how git was designed.

The "agent per branch" concept is actually compelling as a MENTAL MODEL:
- Agent connects → gets a branch (isolated drive)
- Agent writes → commits to its branch
- Agent finishes → merges to main
- Conflicts = two agents edited the same file → merge conflict (detectable, resolvable)

But implementing literal git-on-S3 is messy (git's object model doesn't map well to S3 key-value). And if we're prioritizing semantic search over grep, the "local git for fast grep" benefit diminishes.

**Resolution:** The git MENTAL MODEL (branches for isolation, versions for history, merge for conflict detection) is the right abstraction. Whether the implementation uses actual git or S3 versioning + metadata is a research question for the next phase. Park the implementation, keep the semantics.

### Q: Human UI approach?
**CLI is full from day one** — it's the primary interface for BOTH agents and humans. Web dashboard (admin: drives, permissions, activity) ships incrementally. Drive-style file browser is a later addition. The CLI must cover everything: POSIX ops, drive/permission management, search, version history, activity feed.

**Insights:** This aligns with the agentmail pattern — their CLI outputs JSON for machine parsing, same tool for agents and humans. The CLI becomes the single source of truth for what the product can do. MCP server and web UI are projections of the same capabilities.

### AgentMail Research Findings (Background Research)

Key patterns from agentmail.to relevant to agent-fs:

**Three-surface architecture:** AgentMail exposes the SAME capabilities through Skills (teach agent the SDK), MCP (structured tool calls), CLI (shell commands). All are thin wrappers over one REST API. Build the API first, then wrap it.

**Identity = Resource:** The agent's identity IS an inbox it creates. It doesn't log in as "Agent X" — it creates an inbox and operates through it. For agent-fs: the agent's identity could be tied to an org/drive it provisions.

**Trust escalation:** Agent inboxes can only send 10 emails/day unless authenticated by a human. This is a compelling pattern — agents start constrained, earn expanded capabilities through human endorsement. For agent-fs: agents could have constrained write quotas by default, expandable when a human vouches.

**Pods for tenant isolation:** Hard isolation boundaries between customers/agents. Maps to agent-fs orgs.

**Flat CRUD tool naming:** `{verb}_{noun}` — `list_inboxes`, `get_message`, `send_message`. Simple, predictable. Selective tool loading via `--tools` flag for least-privilege.

**Machine-readable by default:** CLI outputs JSON, not pretty tables. Right call for agent-facing tools.

**Drafts as human-in-the-loop:** Agents create drafts, humans approve before sending. For agent-fs: "staged writes" — agent proposes a file change, human approves before commit.

**API key at account level, identity at resource level:** One API key authenticates the account. Inboxes (identities) are created/managed as resources under that account.

**Pricing reference:** Free (3 inboxes), Dev ($20/mo, 10 inboxes), Startup ($200/mo, 10 pods), Enterprise (custom). YC S25, $6M seed from General Catalyst.

### Q: Trust model and onboarding?
**Google Workspace model:** New agents either CREATE an org or JOIN one (via email-based invitation). On registration, every user gets a **personal org** (auto-created, single-member). They can then create additional orgs or be invited to existing ones.

Each org has:
1. **Default drive** — the main shared drive in the org
2. **Additional drives** — created as needed (e.g., per-project)

Personal files live in the **default drive of your personal org** — NOT cross-org. To share files with another org, share them explicitly.

**The model maps to:**
- agent-fs **Org** ← Google Workspace org (personal org ≈ My Drive, shared org ≈ Shared Drive)
- agent-fs **Drive** ← Google Shared Drive (per-project/team, RBAC controlled)
- agent-fs **Personal org's default drive** ← Google My Drive (private to you)
- agent-fs **Identity** ← email address (like agentmail)

**Insights:** This is clean and consistent with the `/<org_id>/` path structure. Everything lives under an org — even personal files. The personal org is just a single-member org you always own. No special "personal/" prefix or cross-org magic needed. The email-as-identity choice means inviting an agent to an org is the same as inviting a human — send them an email. S3 path structure: `/<org_id>/drives/<drive_id>/path/to/file.md`.

### Q: API surface architecture?
**REST API is the foundation.** MCP server and CLI are AUTO-GENERATED from the REST API spec. One source of truth (the API), two derived interfaces.

**Insights:** This is the agentmail pattern and it's the right call. If you define your REST API with an OpenAPI spec, you can generate:
- MCP tool definitions from the endpoint schemas
- CLI commands from the endpoint structure (resource + action = subcommand)
- Client SDKs (TypeScript, Python) from the spec
- Documentation from the spec

This means the API spec IS the product definition. Changes to the API automatically propagate to MCP + CLI. No drift, no manual sync. Tools like Fern (which agentmail uses) or custom codegen can handle this.

### Q: Embedding pipeline design?
**Pluggable with provider selection at setup.** User configures their preferred embedding provider during initial setup:
- OpenAI (text-embedding-3-small/large)
- Google Gemini (new embedding model)
- Local model (nomic-embed, or similar)

Each provider has different vector dimensions, so this is a one-time setup choice (changing provider means re-embedding everything). Configuration stored alongside the agent-fs instance config.

**Pipeline:** file write → detect content type → extract text → chunk → embed via configured provider → store vectors in SQLite-vec.

**Resolved (see Core Requirement #10 and `thoughts/taras/research/2026-03-14-embeddings-qmd-gemini.md`):**
- qmd uses SQLite + sqlite-vec + FTS5, embeddinggemma-300M (768d), ~900 token chunks with 15% overlap
- Gemini Embedding 2 launched March 2026 — MTEB-leading at 68.32, 768d sweet spot
- Standard dimension: 768 across all providers
- Re-embedding: yes, via `agentfs reindex --force`

### Q: CLI command structure?
**Hybrid:** POSIX ops are top-level, admin ops are namespaced.

```
# File operations (top-level, feels like a real filesystem)
agentfs ls /reports/
agentfs cat /reports/q1.md
agentfs grep "revenue" /reports/
agentfs write /reports/q2.md < content
agentfs edit /reports/q1.md --old "draft" --new "final"
agentfs mv /reports/q1.md /archive/q1.md
agentfs rm /archive/old.md
agentfs search "authentication patterns"
agentfs log /reports/q1.md
agentfs diff /reports/q1.md --v1 3 --v2 5
agentfs recent / --since 1h
agentfs stat /reports/q1.md

# Admin operations (namespaced)
agentfs drive create my-project
agentfs drive list
agentfs drive invite agent@example.com --role editor
agentfs auth login
agentfs auth whoami
agentfs config set embedding-provider openai
```

**Insights:** This gives agents the most intuitive experience (POSIX at the top) while keeping admin ops organized. The `--json` flag on every command enables machine parsing (agentmail pattern).

## Synthesis

### Key Decisions

1. **Product, not internal tool** — agent-fs is a developer product targeting any MCP-capable agent harness (Claude Code, Codex, Pi, etc.). Multi-tenancy and auth are first-class from day one.

2. **Agents and humans are equals** — No separate identity classes. Both are principals in the system with the same capabilities. Email-based identity (borrowing from agentmail).

3. **Google Drive model** — Org → Drives. Every user gets a personal org on registration (with a default drive). Users join additional orgs via invitation and access shared drives within them. RBAC: viewer, editor, admin per drive.

4. **S3 as storage backbone** — S3-compatible object storage (AWS S3, R2, MinIO) is the source of truth. Paths ARE the organizational structure (no separate folder abstraction).

5. **POSIX-named API, not full POSIX compliance** — Tools use literal POSIX names (`ls`, `cat`, `grep`, `mv`, `cp`, `rm`) for zero agent learning curve. Simplified semantics (no file locks, no byte-level random access). FUSE mount is a future layer.

6. **REST API is the foundation** — MCP server and CLI are auto-generated from the OpenAPI spec. One source of truth, zero drift.

7. **Hybrid CLI structure** — POSIX ops top-level (`agentfs ls`), admin ops namespaced (`agentfs drive create`). JSON output by default for machine parsing.

8. **Auto-versioning with semantic diffs** — Every write/edit creates a version. `edit(old_string, new_string)` captures the semantic patch. Optional commit messages. S3 versioning handles storage, metadata DB adds the semantic layer.

9. **Hybrid caching** — Single-file reads go direct to S3 (~50-200ms). Multi-file queries (`grep`, `search`) hit local SQLite FTS5 + sqlite-vec indexes. Writes go to S3 + async index update.

10. **Pluggable embedding** — Choose provider at setup (OpenAI, Gemini, local). Content-type-aware chunking. Stored in sqlite-vec.

11. **OSS first, SaaS later** — Core is open source (BYO S3). Hosted SaaS as a managed tier later. Deployment templates for Docker, Fly.io, Railway.

12. **Git semantics on S3 versioning + SQLite** — Build git-like features (log, diff, revert, blame) on S3 native versioning + SQLite metadata. NOT actual git. Branching deferred to v2. (Research: `thoughts/taras/research/2026-03-14-git-versioning-on-s3.md`)

### Open Questions

1. **Git as implementation vs. metaphor** — RESOLVED. ### Constraints Identified

1. **Zero external dependencies for OSS** — Must run as a single process with embedded storage (SQLite). No Postgres, no Redis, no Qdrant in the default setup.

2. **S3-compatible** — Must work with any S3-compatible storage (AWS S3, Cloudflare R2, MinIO, Backblaze B2), not just AWS.

3. **Embedding provider lock-in** — Vector dimensions differ per provider; switching means full re-embedding. Must be a setup-time decision.

4. **SQLite scaling ceiling** — SQLite works for single-node OSS but won't scale for multi-node SaaS. Need an abstraction layer from day one so SQLite can be swapped for Postgres+pgvector later.

5. **Agent context window limits** — All responses must be agent-safe: large file reads need truncation/warnings, search results need to be concise, `ls` must include size metadata.

6. **Eventual consistency on index** — Async indexing means brief windows where freshly written files aren't searchable. This is acceptable but must be communicated clearly.

### Core Requirements

1. **File operations:** `ls`, `cat`, `write`, `edit`, `append`, `rm`, `mv`, `cp`, `head`, `tail`, `stat`, `mkdir`
2. **Search:** `search` (semantic via embeddings), `grep` (full-text via FTS5)
3. **Versioning:** `log`, `diff`, `revert` — auto-version on every mutation
4. **Auth & Registration:** `agentfs auth register <email>` → email verification → account created → API key generated. `agentfs auth login` for returning users. API key based for all subsequent calls.
5. **Identity:** Email-based, agents and humans are the same entity type. On registration, a **default personal org** is always auto-created for the user. Users can then create additional orgs or be invited to existing ones.
6. **Organization:** Orgs → Shared Drives (RBAC) + Personal Drives (cross-org)
7. **Permissions:** RBAC with viewer/editor/admin roles per drive
8. **API:** REST API (OpenAPI spec) as foundation, auto-generated MCP server + CLI
9. **Storage:** S3-compatible backend, SQLite for metadata/FTS/embeddings
10. **Embedding:** Pluggable provider, auto-embed on write. Architecture modeled on qmd (github.com/tobi/qmd):
    - **Providers:** OpenAI text-embedding-3-small ($0.02/MTok), Gemini embedding-001 ($0.15/MTok, best quality/price), local embeddinggemma-300M (free, 768d)
    - **Standard dimension:** 768 (universal sweet spot across all providers)
    - **Chunking:** ~900 tokens per chunk, 15% overlap, smart markdown boundary detection (from qmd)
    - **Storage:** sqlite-vec for vectors, FTS5 for full-text (both in single SQLite DB)
    - **Pipeline:** write → detect content type → extract text → chunk → embed → index (async, non-blocking)
    - **Re-embed:** `agentfs config set embedding-provider <provider>` + `agentfs reindex --force` (required on provider switch)
    - Full research: `thoughts/taras/research/2026-03-14-embeddings-qmd-gemini.md`
11. **Distribution:** OSS (BYO S3), deployment templates, SaaS tier later
12. **DX:** JSON output by default, rich error messages, agent-safe response sizes

### v1.5 Requirements

13. **Local file linking (sync):** Ability to link local files/folders to agent-fs paths. A daemon or watcher syncs changes bidirectionally (or with a manual `agentfs sync` command). Similar to Dropbox selective sync or Google Drive desktop client.

- Embedding model cached at `~/.agentfs/models/`

### Option 2: Hosted Mode (SaaS)

**Architecture: proxy gateway + per-org worker machines**

Yes — two layers:

1. **Gateway API** (`api.agentfs.dev`) — a lightweight, always-on proxy that handles:
   - Auth (validate API key → resolve to user → resolve to org)
   - Routing (`/<org_id>/*` → correct org worker)
   - User/org management (registration, org creation, invitations — these are global, not per-org)
   - Fly Machine lifecycle (spin up/down org workers on demand)

2. **Org Worker Machines** (one per org, internal only) — the actual agent-fs logic:
   - Own SQLite DB (metadata, FTS5, embeddings)
   - Own S3 credentials (either BYO or Tigris-provisioned)
   - Handles all file ops (`cat`, `write`, `search`, etc.)
   - Never exposed directly to the internet — only reachable via the gateway

```
Internet → api.agentfs.dev (Gateway - always on, lightweight)
              │
              ├── Auth + routing
              │
              ├── POST /orgs/org_1/ops → [Fly internal] → Org Worker A (org_1's SQLite + S3)
              ├── POST /orgs/org_2/ops → [Fly internal] → Org Worker B (org_2's SQLite + S3)
              └── POST /auth/register  → Gateway handles directly (global DB)
```

**Gateway specifics:**
- Tiny: shared-cpu-1x, 256MB. Just proxying + auth.
- Has its own SQLite DB for: user accounts, org membership, API keys, org→machine mapping
- Uses Fly.io's internal DNS (`<app>.internal`) to reach org workers
- Auto-starts org workers on first request (like Fly's auto-start), auto-stops on idle

**Org Worker specifics:**
- Each org = its own Fly Machine with own SQLite + own S3 creds
- Full isolation — no shared state between orgs
- S3 options: BYO S3 (org admin provides creds) or managed Tigris (default for free tier)
- Auto-stop when idle (Fly.io feature) — cost-efficient for low-activity orgs
- LiteFS for replication within an org if needed

**Benefits:**
- Clean security boundary — orgs can never access each other's data
- Gateway is cheap (one small always-on machine) — org workers scale to zero
- Easy to migrate: move the SQLite DB + point to the S3 bucket
- Per-org resource scaling is natural (heavy orgs get bigger workers)

**Local mode comparison:** In local mode, there's no gateway — the daemon IS the org worker. Same code, just running directly instead of behind a proxy.

## API Design (Deep Dive)

### Core Pattern: Command API (POST-based RPC)

Not RESTful — intentionally. A single endpoint per org that accepts operations:

```
POST /orgs/{org_id}/ops
Authorization: Bearer <api_key>
Content-Type: application/json

{ "op": "<operation>", ...params }
```

This maps 1:1 to CLI commands and MCP tools. Auto-generation is trivial: each `op` schema becomes one CLI subcommand and one MCP tool.

### File Operations

```jsonc
// ls
{ "op": "ls", "path": "/reports/", "drive": "main" }
→ { "entries": [{ "name": "q1.md", "type": "file", "size": 2048, "author": "agent@co.com", "modified": "2026-03-14T10:00:00Z" }, ...] }

// cat (with offset/limit for large files)
{ "op": "cat", "path": "/reports/q1.md", "drive": "main", "offset": 0, "limit": 200 }
→ { "content": "...", "total_lines": 450, "truncated": true }

// write (creates or overwrites, creates new version)
{ "op": "write", "path": "/reports/q2.md", "drive": "main", "content": "...", "message": "Initial Q2 report" }
→ { "version": 1, "path": "/reports/q2.md", "size": 1024 }

// edit (semantic patch, creates new version)
{ "op": "edit", "path": "/reports/q1.md", "drive": "main", "old_string": "draft", "new_string": "final", "message": "Finalized Q1" }
→ { "version": 4, "path": "/reports/q1.md", "changes": 1 }

// append
{ "op": "append", "path": "/logs/agent.log", "drive": "main", "content": "2026-03-14 task completed\n" }
→ { "version": 12, "size": 4096 }

// stat (metadata without content)
{ "op": "stat", "path": "/reports/q1.md", "drive": "main" }
→ { "path": "/reports/q1.md", "size": 2048, "type": "text/markdown", "author": "agent@co.com", "created": "...", "modified": "...", "version": 4, "embedded": true }

// head / tail
{ "op": "head", "path": "/reports/q1.md", "drive": "main", "lines": 20 }
{ "op": "tail", "path": "/reports/q1.md", "drive": "main", "lines": 20 }

// rm, mv, cp, mkdir
{ "op": "rm", "path": "/reports/old.md", "drive": "main" }
{ "op": "mv", "from": "/reports/q1.md", "to": "/archive/q1.md", "drive": "main" }
{ "op": "cp", "from": "/reports/q1.md", "to": "/backup/q1.md", "drive": "main" }
{ "op": "mkdir", "path": "/new-folder/", "drive": "main" }
```

### Search Operations

```jsonc
// search (semantic)
{ "op": "search", "query": "authentication patterns", "drive": "main", "limit": 10 }
→ { "results": [{ "path": "/docs/auth.md", "score": 0.92, "snippet": "...", "author": "...", "modified": "..." }, ...] }

// grep (full-text via FTS5)
{ "op": "grep", "pattern": "TODO", "path": "/src/", "drive": "main" }
→ { "matches": [{ "path": "/src/api.ts", "line": 42, "content": "// TODO: add rate limiting" }, ...] }
```

### Versioning Operations

```jsonc
// log (version history)
{ "op": "log", "path": "/reports/q1.md", "drive": "main", "limit": 10 }
→ { "versions": [{ "version": 4, "author": "agent@co.com", "timestamp": "...", "op_type": "edit", "message": "Finalized Q1", "diff_summary": "draft → final" }, ...] }

// diff
{ "op": "diff", "path": "/reports/q1.md", "drive": "main", "v1": 3, "v2": 4 }
→ { "changes": [{ "type": "replace", "old": "draft", "new": "final", "line": 12 }] }

// revert
{ "op": "revert", "path": "/reports/q1.md", "drive": "main", "version": 3, "message": "Reverting to v3" }
→ { "version": 5, "reverted_to": 3 }
```

POST /auth/register     { "email": "..." }           → sends magic link to email
POST /auth/verify       { "email": "...", "token": "..." } → verifies magic link, returns API key
POST /auth/magic-link   { "email": "..." }           → sends new magic link (returning users)

### MCP Server: Same Binary, Config-Driven

```jsonc
// .mcp.json — local mode
{
  "agentfs": {
    "command": "agentfs",
    "args": ["mcp"],
    "env": { "AGENTFS_API_URL": "http://localhost:7433" }
  }
}

// .mcp.json — hosted mode
{
  "agentfs": {
    "command": "agentfs",
    "args": ["mcp"],
    "env": {
      "AGENTFS_API_URL": "https://api.agentfs.dev",
      "AGENTFS_API_KEY": "af_..."
    }
  }
}
```

`agentfs mcp` starts a stdio MCP server. It always talks to the REST API — in local mode that's `localhost:7433` (the daemon), in hosted mode that's `api.agentfs.dev`. Same binary, same code, different config. The MCP tools are auto-generated from the same op schemas that define the REST API.

**HTTP-based MCP (Streamable HTTP transport):**

For environments that can't run a local binary (claude.ai, web-based agents, remote MCP clients), agent-fs also exposes an HTTP-based MCP endpoint:

```bash
# Local: start an HTTP MCP server alongside the daemon
agentfs mcp --http --port 7434

# Hosted: the gateway exposes it directly
# https://api.agentfs.dev/mcp — Streamable HTTP MCP endpoint
```

```jsonc
// claude.ai or any remote MCP client config
{
  "agentfs": {
    "url": "https://api.agentfs.dev/mcp",
    "headers": {
      "Authorization": "Bearer af_..."
    }
  }
}
```

This means agent-fs works in three modes:
1. **stdio MCP** — for Claude Code, Cursor, etc. (local binary, `agentfs mcp`)
2. **HTTP MCP** — for claude.ai, web agents, remote clients (`agentfs mcp --http` or hosted endpoint)
3. **REST API** — for any HTTP client, SDKs, custom integrations

All three expose the same tools from the same op schemas. Zero divergence.

### Drive Context & Path Resolution

**Active drive:** There's always an active drive context. It defaults to the personal org's default drive. Agents can switch or let paths auto-resolve.

**Path resolution rules:**
1. **Explicit drive path** — `/drive/<drive_id>/reports/q1.md` → uses that drive directly, regardless of active context
2. **Relative path** — `/reports/q1.md` → resolves against the active drive
3. **Switch active drive** — `agentfs drive use <drive_id>` sets the active drive for subsequent calls

```jsonc
// Check current context
{ "op": "context" }
→ { "org": "my-org", "drive": "main", "drive_id": "drv_abc123", "role": "editor" }

// Switch active drive
{ "op": "context", "drive": "research" }
→ { "org": "my-org", "drive": "research", "drive_id": "drv_def456", "role": "viewer" }

// Explicit drive path — no context switch needed
{ "op": "cat", "path": "/drive/drv_def456/notes/ideas.md" }
→ reads from the "research" drive without changing active context

// Relative path — uses active drive
{ "op": "cat", "path": "/reports/q1.md" }
→ reads from whatever drive is currently active
```

**CLI:**
```bash
agentfs drive use research        # switch active drive
agentfs drive current             # show active drive + org + role
agentfs cat /reports/q1.md        # uses active drive
agentfs cat /drive/research/notes/ideas.md  # explicit, no switch
```

**MCP:** The active drive is stored server-side per API key session (or in local config for local mode). Agents don't need to pass `drive` on every call if they've set their context.

### Error Response Format

All errors follow a standard shape — designed for agent consumption (actionable, specific):

```jsonc
// Permission error — tells the agent what they CAN do about it
{
  "error": {
    "code": "PERMISSION_DENIED",
    "message": "You have 'viewer' role on drive 'main'. To write, ask an admin to grant 'editor' access.",
    "op": "write",
    "path": "/reports/q2.md",
    "drive": "main",
    "your_role": "viewer",
    "required_role": "editor"
  }
}

// Not found
{
  "error": {
    "code": "NOT_FOUND",
    "message": "File '/reports/q99.md' does not exist in drive 'main'.",
    "op": "cat",
    "path": "/reports/q99.md",
    "suggestion": "Run 'ls /reports/' to see available files."
  }
}

// Edit conflict (old_string not found)
{
  "error": {
    "code": "EDIT_CONFLICT",
    "message": "The text 'draft figures' was not found in '/reports/q1.md'. The file may have been modified since you last read it.",
    "op": "edit",
    "path": "/reports/q1.md",
    "current_version": 5,
    "suggestion": "Re-read the file with 'cat' and retry the edit."
  }
}

// Indexing in progress (write-then-search consistency)
{
  "error": {
    "code": "INDEXING_IN_PROGRESS",
    "message": "File '/reports/q2.md' was written 2s ago and is still being indexed. It will be searchable shortly.",
    "op": "search",
    "path": "/reports/q2.md",
    "retry_after_ms": 3000
  }
}
```

**Key principles:**
- Every error includes `code` (machine-parseable) + `message` (human/agent-readable)
- Include `suggestion` when there's an actionable next step
- Permission errors include current role + required role
- Edit conflicts include current version for retry logic

### Design Rationale

- **Command API for file ops** → one endpoint, many operations. Maps trivially to MCP tools and CLI commands.
- **REST for admin ops** → standard CRUD for resources that ARE resources (orgs, drives, members). These don't need POSIX naming.
- **Drive defaults to personal** → if `drive` param is omitted, uses the user's personal drive. Reduces boilerplate for simple operations.
- **All responses include metadata** → every response from file ops includes enough info that the agent doesn't need a follow-up `stat` call.
